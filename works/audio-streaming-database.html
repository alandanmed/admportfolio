Overview

The Audio Streaming Database System is a relational database project designed for an audio streaming platform. The database modeled core platform features such as users, creators, songs, albums, podcasts, playlists, subscriptions, streaming activity, and social interactions.

The goal was to design a database that could support real application workflows, not just store isolated records. I worked through the process of translating platform requirements into relational tables, constraints, relationships, views, and stored procedures.

This project helped me understand how backend systems depend on strong data modeling. A clean schema makes it easier to support features, avoid duplicate data, protect data integrity, and write useful queries for analytics or application behavior.

What I Built

I designed and implemented a MySQL database for an audio streaming service. The system included normalized relational tables, primary and foreign key constraints, update and delete rules, SQL views, and stored procedures for platform actions.

I also built queries and database logic to support common streaming platform workflows, such as managing playlists, tracking streaming activity, handling subscriptions, uploading content, and summarizing creator or listener data.

Key Features

• Relational schema for users, creators, songs, albums, podcasts, playlists, and subscriptions
• Primary keys, foreign keys, and constraints to protect data integrity
• Normalized table design to reduce duplication and improve maintainability
• SQL views for playlist summaries, creator statistics, podcast analytics, and streaming activity
• Stored procedures for renewing subscriptions, duplicating playlists, uploading songs, and merging playlists
• Query logic to support platform reporting and user-facing database operations

Impact

This project gave me hands-on experience designing a database that could support a realistic software product. It strengthened my ability to think through relationships between entities, enforce data consistency, and write SQL that supports both application features and analytics.

It also helped me better understand how database design decisions affect the rest of a system. A well-designed schema makes backend logic cleaner, queries easier to maintain, and product features easier to extend.
