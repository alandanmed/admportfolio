Virtual Memory Simulator 💾

Overview

The Virtual Memory Simulator project focused on implementing the core pieces of a virtual memory system in C. The simulator modeled how an operating system translates virtual addresses, manages physical memory frames, handles page faults, and chooses pages to evict when memory is full.

This project helped me understand what happens behind the scenes when a program accesses memory. Instead of thinking of memory as one large space, I worked with page tables, frame tables, page replacement policies, dirty bits, reference bits, and memory access statistics.

The goal was to simulate how operating systems manage memory efficiently while balancing correctness, performance, and limited physical resources.

What I Built

I implemented the main virtual memory logic for address translation, page table management, page fault handling, frame allocation, and eviction behavior. The simulator had to correctly update memory metadata while responding to reads and writes from simulated processes.

I also implemented page replacement behavior using policies such as Clock Sweep and Approximate LRU, then used simulator statistics to evaluate performance through metrics like memory accesses, page faults, writebacks, and AMAT.

Key Features

• Virtual address translation through page tables
• Page fault detection and handling
• Frame allocation and eviction when physical memory is full
• Clock Sweep page replacement policy
• Approximate LRU page replacement behavior
• Dirty bit and reference bit tracking
• Memory access statistics and AMAT performance analysis

Impact

This project strengthened my understanding of operating systems and low-level memory management. It showed me how much work happens between a memory access and the actual data being loaded or stored.

It also gave me more practice debugging C code where pointer logic, struct fields, and state updates all have to be handled carefully. Small mistakes in virtual memory logic can create major correctness issues, so this project helped me become more disciplined with systems-level debugging.
